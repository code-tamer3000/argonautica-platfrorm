# Лог решений (ADR-lite)

Хронологический список принятых архитектурных решений. Дополнять по ходу проекта.
Формат: дата — решение — кратко почему.

## Базовая архитектура
- Стек: PostgreSQL + Redis + FastAPI (REST + WebSocket) + React/Vite + nginx,
  всё в Docker Compose. Причина: знаком разработчику, ложится на задачу.
- API: REST для «холодного», WebSocket для реалтайма. Чистый REST не пушит.
- Монорепо (backend + frontend вместе). Причина: проще для соло-разработки.

## База данных
- Postgres вместо SQLite — конкурентная запись под чат с тредами.
- id — BIGSERIAL (монотонный), не UUID. Причина: на монотонности держится
  механизм статусов прочтения.
- Треды одноуровневые (thread_root_id, привязка к корню). Стиль Slack.
- Каналы — вариант А (неявный доступ, ленивые строки room_members).
- Статусы прочтения — через last_read_message_id, без таблицы прочтений.
- Мягкое удаление сообщений (deleted_at).
- Медиа централизованы в media_assets (абстракция над хранилищем).

## Реалтайм
- Redis подключён с первого дня; доставка через pub/sub изначально (не зависит от
  числа воркеров). Redis также: presence, «печатает», сессии, rate-limit.
- Эфемерное состояние («печатает», presence) — только в Redis, не в Postgres.

## Аутентификация
- JWT (access + refresh). Access короткий; refresh — списком в Redis ради
  мгновенного отзыва (логаут устройства, бан).

## Хранилище медиа
- MinIO (S3-совместимо), приватные бакеты, доступ через presigned URL после
  проверки прав. Загрузка/чтение напрямую клиент↔MinIO (presigned-PUT/GET).
- Причина выбора: self-hosted, переезд на managed-S3 потом почти без правок кода.
- Помнить: MinIO лежит на том же диске — ёмкость это не увеличивает.

## Деплой
- Blue-green (zero-downtime). Stateful-сервисы (Postgres/Redis/MinIO) общие.
- Следствие: миграции только обратно-совместимые (expand/contract).
- WebSocket: при переключении сокеты рвутся — клиент переподключается.

## Быстрая доставка медиа (превью + presigned в ленте + кэш)
Симптом: фото/видео в ленте грузятся медленно, особенно с телефона, хотя сервер мощный,
а файлы «копеечные». Причина — не размер файлов и не железо, а **латентность и число
round-trip'ов**, которые мобильная сеть жёстко усиливает. Три меры (каждая — отдельный
слой):

- **Presigned-URL в payload сообщений**, а не по запросу на каждый ассет. Раньше каждое
  вложение = `GET /api/media/{id}` (проверка прав + БД) → затем сам файл, на отдельном
  домене (DNS+TLS). Экран с 15 фото = ~30 последовательных запросов ещё до первого
  байта; при пинге 150 мс это секунды. Теперь `MessageOut.attachments` несёт готовые
  ссылки (батч-подпись, локальна). Доступ гейтится комнатой (кто читает сообщение —
  читает вложения), поэтому per-asset проверка не нужна — как и для аватаров/стикеров.
  `attachment_ids` оставлен для обратной совместимости.
- **Превью в ленте** вместо мегабайтных оригиналов (десятки КБ), оригинал — по клику.
  `media_assets.thumb_key`, генерация best-effort (сбой не роняет загрузку). Два пути:
  - *картинки* — сервер (Pillow, WebP, ≤1024px) один раз при подтверждении тянет
    оригинал из MinIO и ужимает. Это **единственное** место, где байты проходят через
    бэкенд, и то раз на загрузку, не на просмотр — принцип п.7 для просмотра сохранён.
  - *видео* — постер снимает **клиент** (canvas → WebP) и льёт отдельным объектом,
    ключ передаёт в `confirm` (`thumb_storage_key`). Сервер видеофайл ради кадра не
    тянет (большой файл → дорого, против п.7). Защита от подмены: ключ постера обязан
    иметь живое намерение загрузки того же пользователя вида image (иначе `thumb_key`
    можно было бы навести на чужой объект) — не сходится → видео просто без постера.
    Тот же постер идёт в `poster` у `<video>`: нет чёрного прямоугольника до плея.
- **Backfill старых картинок**: превью появились не сразу, у ранее загруженных фото
  `thumb_key` = NULL → грузились оригиналом. Разовый `scripts/backfill_thumbnails.py`
  досоздаёт им превью (идемпотентно, пачками). Видео не трогает — их постер бывает
  только с клиента при загрузке; у старых видео превью просто не будет.
- **Кэш в браузере**: presigned-GET TTL поднят до 24 ч (SigV4 позволяет до 7 дней; для
  ~20 доверенных участников приемлемо) + nginx отдаёт `Cache-Control: private,
  max-age=86400, immutable` на медиа. Объекты неизменяемы (ключ = uuid), поэтому в
  пределах жизни ссылки байты не перекачиваются при перерисовке/скролле. Плюс `gzip` на
  текстовые ответы (SPA-бандл, JSON); медиа не жмём — уже сжато.

Про «без VPN быстро, с телефона медленно» — вне репозитория (сетевой/гео-фактор:
латентность + возможный шейпинг у оператора); меры выше бьют именно по латентности.

## Вне MVP (на вырост)

- Категории базы знаний (kb_categories).
