# Лог решений (ADR-lite)

Хронологический список принятых архитектурных решений. Дополнять по ходу проекта.
Формат: дата — решение — кратко почему.

## Базовая архитектура
- Стек: PostgreSQL + Redis + FastAPI (REST + WebSocket) + React/Vite + nginx,
  всё в Docker Compose. Причина: знаком разработчику, ложится на задачу.
- API: REST для «холодного», WebSocket для реалтайма. Чистый REST не пушит.
- Монорепо (backend + frontend вместе). Причина: проще для соло-разработки.

## База данных
- Postgres вместо SQLite — конкурентная запись под чат с тредами.
- id — BIGSERIAL (монотонный), не UUID. Причина: на монотонности держится
  механизм статусов прочтения.
- Треды одноуровневые (thread_root_id, привязка к корню). Стиль Slack.
- Каналы — вариант А (неявный доступ, ленивые строки room_members).
- Статусы прочтения — через last_read_message_id, без таблицы прочтений.
- Мягкое удаление сообщений (deleted_at).
- Медиа централизованы в media_assets (абстракция над хранилищем).

## Разделы участника
- **«Каюта»** (личная психо-проработка): три подраздела (дневник/декатастрофизация/
  триггеры) в одной таблице `cabin_entries`, поля формы — в JSONB `data` под
  `kind`-дискриминатор. Причина: формы отличаются полями, но не структурно; JSONB
  даёт менять поля без миграций и без трёх таблиц-близнецов. Приватна: видит/правит/
  удаляет только автор; админ — read-only просмотр (`/api/cabin/admin/...`). Удаление
  записей физическое (исключение из мягкого): восстанавливать личную заметку незачем.
- **«Динамика»** (трекинг ежедневного ДЗ, 28 дней, категории focus/notes/film):
  записи ДЗ — это `messages` в личной комнате участника (`rooms.is_personal`),
  отдельной таблицы записей нет. Причина: ДЗ и так постится в чат-дневник, дублировать
  сущность незачем; прогресс (закрытые дни/стрик/просрочки) считается на лету из
  сообщений. Исключения хранятся отдельно: `journal_pardons` (участник прощает себе
  день, лимит 3) и `journal_credits` (админ вручную зачитывает день, без лимита).
- **Новостной канал** — `rooms.is_news` (singleton, создаётся в lifespan
  `ensure_news_channel`); верхнеуровневые посты — только admin. Репост в новости —
  копия исходного сообщения с `forwarded_from_sender_id` (атрибуция «переслано от X»),
  а не ссылка: пост живёт независимо от оригинала.
- **Уведомления** (колокольчик) — в Postgres (`notifications`), не в Redis: нужна
  история, переживание перезагрузки и будущий web-push. Типы: `dm`/`reply`/`news`
  (от сообщений) + системный `journal_missed` (вчерашний день ДЗ не закрыт).
  Генерируются в транзакции отправки; realtime — персональный канал `user:{id}` в
  Redis pub/sub (WS `notification.new`/`notification.removed`).
- **«Поддержка»** — обращения (`feedback`: improvement/bug, со статусом `resolved_at`)
  и FAQ (`faq_items`, ведёт admin). В Postgres — нужна история и разбор в «Управлении».
- **Комментарии базы знаний** (`kb_comments`) — плоские, мягкое удаление (автор/admin).
- **Голосовые сообщения** — новый `media_assets.kind = 'audio'`, тем же media-flow
  (presigned-PUT/GET), без отдельной сущности.

## Тех-мониторинг
- Админ-метрики сервера (`/api/admin/metrics`) читаются напрямую из `/proc` — без
  внешних зависимостей; мгновенные скорости (CPU %, сеть) — дельта между снимками,
  предыдущий снимок в Redis (`admin:metrics:prev`), корректно между воркерами.

## Реалтайм
- Redis подключён с первого дня; доставка через pub/sub изначально (не зависит от
  числа воркеров). Redis также: presence, «печатает», сессии, rate-limit.
- Эфемерное состояние («печатает», presence) — только в Redis, не в Postgres.

## Аутентификация
- JWT (access + refresh). Access короткий; refresh — списком в Redis ради
  мгновенного отзыва (логаут устройства, бан).

## Хранилище медиа
- MinIO (S3-совместимо), приватные бакеты, доступ через presigned URL после
  проверки прав. Загрузка/чтение напрямую клиент↔MinIO (presigned-PUT/GET).
- Причина выбора: self-hosted, переезд на managed-S3 потом почти без правок кода.
- Помнить: MinIO лежит на том же диске — ёмкость это не увеличивает.

## Деплой
- Blue-green (zero-downtime). Stateful-сервисы (Postgres/Redis/MinIO) общие.
- Следствие: миграции только обратно-совместимые (expand/contract).
- WebSocket: при переключении сокеты рвутся — клиент переподключается.

## Вне MVP (на вырост)
- Пуш-уведомления (Web Push).
- Категории базы знаний (kb_categories).

## Открытые вопросы
- Хранение токенов на фронте (httpOnly-cookie + CSRF).
- На чём переключать blue/green (вероятно nginx + reload).
- Формат стикеров и кто добавляет паки.
