# Лог решений (ADR-lite)

Хронологический список принятых архитектурных решений. Дополнять по ходу проекта.
Формат: дата — решение — кратко почему.

## Базовая архитектура
- Стек: PostgreSQL + Redis + FastAPI (REST + WebSocket) + React/Vite + nginx,
  всё в Docker Compose. Причина: знаком разработчику, ложится на задачу.
- API: REST для «холодного», WebSocket для реалтайма. Чистый REST не пушит.
- Монорепо (backend + frontend вместе). Причина: проще для соло-разработки.

## База данных
- Postgres вместо SQLite — конкурентная запись под чат с тредами.
- id — BIGSERIAL (монотонный), не UUID. Причина: на монотонности держится
  механизм статусов прочтения.
- Треды одноуровневые (thread_root_id, привязка к корню). Стиль Slack.
- Каналы — вариант А (неявный доступ, ленивые строки room_members).
- Статусы прочтения — через last_read_message_id, без таблицы прочтений.
- Мягкое удаление сообщений (deleted_at).
- Медиа централизованы в media_assets (абстракция над хранилищем).

## Разделы участника
- **«Каюта»** (личная психо-проработка): три подраздела (дневник/декатастрофизация/
  триггеры) в одной таблице `cabin_entries`, поля формы — в JSONB `data` под
  `kind`-дискриминатор. Причина: формы отличаются полями, но не структурно; JSONB
  даёт менять поля без миграций и без трёх таблиц-близнецов. Приватна: видит/правит/
  удаляет только автор; админ — read-only просмотр (`/api/cabin/admin/...`). Удаление
  записей физическое (исключение из мягкого): восстанавливать личную заметку незачем.
- **«Динамика»** (трекинг ежедневного ДЗ, 28 дней, категории focus/notes/film):
  записи ДЗ — это `messages` в личной комнате участника (`rooms.is_personal`),
  отдельной таблицы записей нет. Причина: ДЗ и так постится в чат-дневник, дублировать
  сущность незачем; прогресс (закрытые дни/стрик/просрочки) считается на лету из
  сообщений. Исключения хранятся отдельно: `journal_pardons` (участник прощает себе
  день, лимит 3) и `journal_credits` (админ вручную зачитывает день, без лимита).
- **Новостной канал** — `rooms.is_news` (singleton, создаётся в lifespan
  `ensure_news_channel`); верхнеуровневые посты — только admin. Репост в новости —
  копия исходного сообщения с `forwarded_from_sender_id` (атрибуция «переслано от X»),
  а не ссылка: пост живёт независимо от оригинала.
- **Уведомления** (колокольчик) — в Postgres (`notifications`), не в Redis: нужна
  история, переживание перезагрузки и будущий web-push. Типы: `dm`/`reply`/`news`
  (от сообщений) + системный `journal_missed` (вчерашний день ДЗ не закрыт).
  Генерируются в транзакции отправки; realtime — персональный канал `user:{id}` в
  Redis pub/sub (WS `notification.new`/`notification.removed`).
- **«Поддержка»** — обращения (`feedback`: improvement/bug, со статусом `resolved_at`)
  и FAQ (`faq_items`, ведёт admin). В Postgres — нужна история и разбор в «Управлении».
- **Комментарии базы знаний** (`kb_comments`) — плоские, мягкое удаление (автор/admin).
- **Голосовые сообщения** — новый `media_assets.kind = 'audio'`, тем же media-flow
  (presigned-PUT/GET), без отдельной сущности.

## Telegram-бот (доступ и поддержка)
- Отдельный сервис `bot` ([backend/scripts/telegram_bot.py](../backend/scripts/telegram_bot.py))
  на том же образе, что backend. Функции: выдача/сброс пароля по совпадению Telegram-
  `@username` с логином + канал техвопросов (пересылка админу, ответ reply — обратно).
- **HTTP Bot API + long-polling `getUpdates`**, не MTProto/webhook. Причина: на
  RU-хостинге IP Telegram часто заблокированы — обходим обычным **SOCKS5/HTTP-прокси**
  (`TELEGRAM_PROXY`), который туннелирует HTTPS httpx; MTProxy не нужен. Webhook не
  берём: не хотим публичный роут и лишний TLS-эндпоинт.
- **Singleton, вне blue-green** и без healthcheck (не слушает :8000): два поллера на
  одном токене конфликтовали бы за `getUpdates`. Поэтому же бота **нет на staging**
  (второй long-polling с прод-токеном сломал бы прод-бота).
- Выдача пароля — это **пере-выдача** (в БД только argon2-хеш, plaintext живёт лишь в
  сообщении), с `must_change_password=true`. Аккаунты заводятся заранее, бот их не
  создаёт.
- **Риск принят:** Telegram-username можно сменить → теоретический захват чужого
  логина, совпавшего по нику. Группа закрытая (~30 чел.), пароль временный и требует
  смены — риск осознанный. Частота выдач ограничена через Redis.
- Эфемерное состояние бота — только в Redis (см. `DATA_MODEL.md`): rate-limit, ожидание
  вопроса, маппинг «сообщение-вопрос у админа → кому доставить ответ».

## Реалтайм
- Redis подключён с первого дня; доставка через pub/sub изначально (не зависит от
  числа воркеров). Redis также: presence, «печатает», сессии, rate-limit.
- Эфемерное состояние («печатает», presence) — только в Redis, не в Postgres.

## Аутентификация
- JWT (access + refresh). Access короткий; refresh — списком в Redis ради
  мгновенного отзыва (логаут устройства, бан).

## Хранилище медиа
- MinIO (S3-совместимо), приватные бакеты, доступ через presigned URL после
  проверки прав. Загрузка/чтение напрямую клиент↔MinIO (presigned-PUT/GET).
- Причина выбора: self-hosted, переезд на managed-S3 потом почти без правок кода.
- Помнить: MinIO лежит на том же диске — ёмкость это не увеличивает.

## Деплой
- Blue-green (zero-downtime). Stateful-сервисы (Postgres/Redis/MinIO) общие.
- Следствие: миграции только обратно-совместимые (expand/contract).
- WebSocket: при переключении сокеты рвутся — клиент переподключается.

## Тестовый стенд (staging)
- Автодеплой из `develop` (workflow `Deploy → staging` на push) на тот же сервер, что
  и прод, но в `/opt/platform-staging` — превью новых фич до релиза в `main`.
- Изоляция от прода — через **отдельный compose-проект** `platform-staging`
  (`docker-compose.staging.yml`): своя docker-сеть, свои контейнеры и свой namespace
  volume'ов. Postgres/Redis/MinIO стенда — новые пустые тома, прод-данные не пересекаются.
- Наружу — только nginx на `8443` (прод держит 80/443). Доступ по `https://<IP>:8443`,
  self-signed TLS. Стенд открыт (без Basic Auth) — внутренний превью. Свой `.env` с
  отдельным `JWT_SECRET`.
- Стенд **без blue-green** (не нужен zero-downtime) и **без сервиса `bot`**: второй
  long-polling с прод-токеном сломал бы прод-бота.

## Каюта — доступ по выдаче (не всем по умолчанию)
- «Каюта» — чувствительная психо-проработка, не всем нужна сразу. Поэтому раздел
  закрыт по умолчанию: право хранится **флагом `users.can_access_cabin`** (default
  `false`), а не ролью — гранулярность на уровне участника, как у `can_create_groups`.
  Отдельную таблицу прав не заводим — одного булева достаточно, дешевле и проще.
- Проверка — серверная зависимость `require_cabin_access` на личных эндпоинтах Каюты
  (п.1: авторизация на каждом запросе). Admin имеет доступ неявно (он же его и
  раздаёт) — чтобы не заставлять админа выдавать право самому себе.
- О выдаче участник узнаёт через **уведомление-колокольчик** `cabin_granted`, а не
  молча: раздел просто «появился бы» в навигации — легко не заметить. Уведомление —
  системное, без комнаты, поэтому `notifications.room_id` сделали **nullable** (вместо
  фиктивной привязки к какой-нибудь комнате); клик ведёт в `/cabin` (роутинг по `kind`
  на фронте, а не по `room_id`).

## Быстрая доставка медиа (превью + presigned в ленте + кэш)
Симптом: фото/видео в ленте грузятся медленно, особенно с телефона, хотя сервер мощный,
а файлы «копеечные». Причина — не размер файлов и не железо, а **латентность и число
round-trip'ов**, которые мобильная сеть жёстко усиливает. Три меры (каждая — отдельный
слой):

- **Presigned-URL в payload сообщений**, а не по запросу на каждый ассет. Раньше каждое
  вложение = `GET /api/media/{id}` (проверка прав + БД) → затем сам файл, на отдельном
  домене (DNS+TLS). Экран с 15 фото = ~30 последовательных запросов ещё до первого
  байта; при пинге 150 мс это секунды. Теперь `MessageOut.attachments` несёт готовые
  ссылки (батч-подпись, локальна). Доступ гейтится комнатой (кто читает сообщение —
  читает вложения), поэтому per-asset проверка не нужна — как и для аватаров/стикеров.
  `attachment_ids` оставлен для обратной совместимости.
- **Серверные превью картинок** (Pillow, WebP, ≤1024px), генерятся один раз при
  подтверждении загрузки и хранятся в `media_assets.thumb_key`. Лента отдаёт превью
  (десятки КБ вместо мегабайтных оригиналов с телефона), оригинал — по клику. Генерация
  best-effort: сбой не роняет загрузку. Это **единственное** место, где байты проходят
  через бэкенд, и то раз на загрузку, не на просмотр — принцип «не гонять файлы через
  FastAPI» (п.7) для просмотра сохранён. Видео-постеры — на вырост (тянуть большой файл
  на бэкенд дорого; кандидат — клиентский кадр при загрузке).
- **Кэш в браузере**: presigned-GET TTL поднят до 24 ч (SigV4 позволяет до 7 дней; для
  ~20 доверенных участников приемлемо) + nginx отдаёт `Cache-Control: private,
  max-age=86400, immutable` на медиа. Объекты неизменяемы (ключ = uuid), поэтому в
  пределах жизни ссылки байты не перекачиваются при перерисовке/скролле. Плюс `gzip` на
  текстовые ответы (SPA-бандл, JSON); медиа не жмём — уже сжато.

Про «без VPN быстро, с телефона медленно» — вне репозитория (сетевой/гео-фактор:
латентность + возможный шейпинг у оператора); меры выше бьют именно по латентности.
