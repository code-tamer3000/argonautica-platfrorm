# Платформа образовательного проекта — Техническая документация

> Статус: проектирование. Версия 0.3.
> Этот документ — единый источник правды по архитектуре и процессу.
> Модель данных вынесена в отдельный документ `DATA_MODEL.md`.

---

## 1. Назначение

Платформа для образовательного проекта. Две главные функции:

- **База знаний** — размещение материалов автора. Контент публикует и модерирует
  админ, участники получают к нему доступ.
- **Пространство для общения** — реалтайм-чат участников: личные сообщения,
  группы, общие каналы, треды под сообщениями.

Масштаб небольшой по числу людей (~20–30 активных участников), но с полноценной
реалтайм-механикой и хранением медиа. Узкое место проекта — не число
пользователей, а хранение и раздача видео (см. раздел 5).

Платформа должна работать и как мобильное веб-приложение (PWA) — см. 4.11.

---

## 2. Технологический стек

| Слой | Технология | Роль |
|------|------------|------|
| База данных | **PostgreSQL** | Основное хранилище: пользователи, сообщения, треды, база знаний, календарь |
| Кэш / шина | **Redis** | Pub/sub между воркерами, presence, «печатает», refresh-токены, rate-limiting |
| Бэкенд | **FastAPI** | REST-эндпоинты + WebSocket для реалтайма |
| ORM | **SQLAlchemy (async)** | Слой между Python и Postgres |
| Миграции | **Alembic** | Управляемые, версионируемые изменения структуры БД |
| Фронтенд | **React + Vite** | SPA-клиент, собираемый как PWA |
| Хранилище медиа | **MinIO** | S3-совместимое объектное хранилище: фото, видео, файлы, стикеры |
| Веб-сервер / прокси | **nginx** | Точка входа, HTTPS, раздача статики, маршрутизация, переключение blue/green |
| Оркестрация | **Docker Compose** | Все сервисы в контейнерах, единое окружение локально и на проде |

**Архитектура API — REST**, дополненная **WebSocket** для живой доставки:
- REST — всё «холодное»: логин, личный кабинет, списки комнат, история сообщений,
  чтение базы знаний.
- WebSocket — всё «живое»: доставка сообщений, обновления тредов, «печатает»,
  presence.

## 3. Архитектура

### 3.1. Топология контейнеров

```
                  Интернет
                     │
                     ▼
            ┌─────────────────┐
            │      nginx       │  ← порты 80/443, HTTPS (Let's Encrypt)
            │ (reverse proxy)  │     статика фронта + переключатель blue/green
            └───┬─────────┬────┘
        /api,/ws│         │/  (статика фронта, PWA)
                ▼         ▼
        ┌──────────────┐  (собранный React + service worker)
        │   FastAPI    │───────────────┐ presigned-PUT/GET
        │  (uvicorn)   │               ▼ (клиент льёт/читает напрямую)
        └──┬────────┬──┘        ┌────────────┐
           │        │          │   MinIO    │  объектное хранилище
           ▼        ▼          │ (S3 API)   │  (фото/видео/файлы)
   ┌────────────┐ ┌────────┐   └────────────┘
   │ PostgreSQL │ │ Redis  │
   └────────────┘ └────────┘
```

> Медиа через MinIO: FastAPI выдаёт клиенту presigned-ссылки, и клиент
> загружает/скачивает файлы напрямую из MinIO, минуя приложение. Это снимает с
> FastAPI прокачку видео.

### 3.2. Путь запроса
Весь внешний трафик сначала попадает в **nginx**. Он расшифровывает HTTPS и по
адресу решает: `/` → статика фронта с диска; `/api/*` → FastAPI; `/ws` → WebSocket
в FastAPI; медиа → отдаёт с диска напрямую. Приложение наружу не торчит — слушает
локально, доступно только через nginx.

### 3.3. Реалтайм-слой (WebSocket + Redis)
FastAPI запускается в несколько воркеров. WebSocket-соединение живёт внутри
конкретного процесса: юзер A на воркере 1, юзер B на воркере 2 — воркер 1 не видит
соединение B, и сообщение A не дойдёт до B.

**Redis в роли pub/sub** решает это: общая шина, через которую любой воркер
рассылает сообщение всем подключённым.

**Решение: Redis подключаем сразу, с первого дня.** Доставка идёт через pub/sub
изначально — даже пока воркер один. Небольшая плата (publish в Redis → обратная
подписка) окупается тем, что система не зависит от числа воркеров: масштабирование
не требует переделки доставки. Redis сразу несёт presence, «печатает»,
refresh-токены и rate-limiting. Поднимается контейнером в compose с самого начала.

### 3.4. Хранение и раздача медиа (MinIO)
Все файлы (фото, видео, файлы базы знаний, стикеры) хранятся в **MinIO** —
S3-совместимом объектном хранилище, поднимаемом контейнером в compose. Метаданные —
в таблице `media_assets`, абстрагирующей хранилище (см. `DATA_MODEL.md`).

- **Доступ — приватный, через presigned URL.** Бакеты не публичны. Бэкенд
  проверяет права юзера на сообщение/материал и выдаёт временную подписанную ссылку.
  Подобрать чужой файл по URL нельзя.
- **Загрузка** (особенно видео) — по **presigned-PUT**: клиент льёт файл напрямую в
  MinIO, минуя FastAPI. Видео не течёт через приложение.
- **Чтение/стриминг** — по **presigned-GET**, с поддержкой HTTP range-запросов
  (перемотка видео).

> **Важно про ёмкость:** MinIO — это интерфейс к хранилищу, а **не дополнительный
> диск**. Байты лежат на тех же 80 ГБ, пока MinIO не получит отдельный том. То есть
> MinIO решает не ёмкость, а переезд: когда диск кончится, endpoint меняется на
> managed-S3 (Selectel, Yandex Object Storage) почти без правок кода — API тот же.
> Масштабироваться первым по-прежнему будет хранилище.

## 4. Функциональные требования

### 4.1. Авторизация и роли
- Регистрация и вход.
- Аутентификация на **JWT** (access + refresh). Access короткоживущий; refresh
  хранится **списком в Redis** ради мгновенного отзыва (логаут устройства, бан).
- Две роли:

| Роль | Права |
|------|-------|
| **Участник** | Доступ к материалам базы знаний, участие в чатах, личный кабинет |
| **Админ** (автор) | Размещение и модерация контента, управление стикерпаками и каналами, доступ ко всему |

### 4.2. Личный кабинет
- У каждого участника есть личный кабинет: профиль (имя, аватар, био) и настройки.
- Ядро полей фиксированное, прочие настройки — в JSONB-поле без миграций.

### 4.3. Сообщения и медиа
- Текстовые сообщения, вложения (фото и видео), стикеры (см. 4.5).
- Несколько вложений на сообщение.
- Доставка в реальном времени через WebSocket.

### 4.4. Треды (ключевая механика)
- Под каждым сообщением можно развернуть обсуждение.
- **Модель одноуровневая (стиль Slack):** ответы привязаны к корневому сообщению,
  без бесконечной вложенности. Реализация — поле `thread_root_id` (см. `DATA_MODEL.md`).

### 4.5. Стикерпаки
- Поддержка стикерпаков в сообщениях; стикерпаки добавляет админ.

### 4.6. Типы пространств общения
| Тип | Кто создаёт | Видимость |
|-----|-------------|-----------|
| Личный чат (DM) | Любой участник | Только два участника |
| Группа | Любой участник | Только приглашённые участники |
| Канал | Только админ | Виден всем |

- Доступ к каналам — **неявный** (участник платформы видит все каналы; членство не
  материализуется на всех). Подробно в `DATA_MODEL.md`.
- **Аватарки чатов** — у групп и каналов своя картинка; у личного чата
  подставляется аватар собеседника.

### 4.7. Закрепление сообщений
- В комнате можно закреплять сообщения (можно несколько).
- Право закрепления — у владельца комнаты / админа.

### 4.8. Индикаторы активности
- **«Печатает…»** — видно, когда участник набирает сообщение. Эфемерное состояние:
  идёт по WebSocket, в Redis с коротким TTL, в БД не пишется.
- **Прочтения** — видно, кто просмотрел сообщение, а кто нет. Выводится из позиции
  чтения (`last_read_message_id`), без отдельной таблицы на каждое прочтение
  (см. `DATA_MODEL.md`).
- **Presence** — кто сейчас онлайн (Redis).

### 4.9. База знаний
- Раздел с материалами автора: статьи (markdown), файлы, видео.
- Контент размещает и модерирует **админ**; участники читают/смотрят.
- Видео раздаётся через nginx (см. 3.4).
- Категории/разделы — **на вырост**; в MVP допустим плоский список.

### 4.10. Календарь проекта
- Общий календарь событий (даты, заголовки, описания).
- **Привязка событий к комнатам/каналам — да** (событие может относиться к
  конкретному пространству или быть общим для проекта).

### 4.11. Мобильное использование (PWA)
Платформа должна работать как **Progressive Web App**: участник открывает сайт в
браузере на iPhone/Android, добавляет на домашний экран и пользуется как отдельным
приложением (standalone, без браузерной обвязки). Не нативное приложение, не сторы.

Требуется: **Web App Manifest** (имя, иконки, `display: standalone`),
**Service Worker** (кэш оболочки, базовый офлайн, обязателен для установки),
**HTTPS** (уже есть). На Vite закрывается плагином (`vite-plugin-pwa`).

**Пуш-уведомления — на вырост, вне MVP.** Открытый WebSocket доставляет сообщения,
только пока приложение активно; для уведомлений при закрытом приложении нужен
отдельный **Web Push** (со своим бэкендом; на iOS — только для установленного PWA
и с ограничениями). Самостоятельная задача, в MVP не входит.

### 4.12. Дизайн-система
- Весь фронтенд строится по **существующей дизайн-системе проекта**.

## 5. Инфраструктура и развёртывание

### 5.1. Целевой сервер (VPS)
- 2 ядра / 4 ГБ RAM / 80 ГБ диск. Для чата на 20–30 человек — с запасом.
- **Узкое место — хранилище видео**, а не нагрузка чата. MinIO лежит на том же
  диске, поэтому масштабируется первым именно диск (см. 3.4).
- Учесть память: при blue-green в момент переключения недолго работают два
  комплекта API одновременно — на 4 ГБ при таком масштабе проходит, но держим в уме.

### 5.2. Docker Compose: dev vs prod
Все сервисы в `docker-compose.yml`, один `docker compose up` поднимает стек.
Контейнеры видят друг друга по имени; данные Postgres — в именованном volume.
- **Dev:** Vite dev-сервер с hot-reload отдельно; Postgres и Redis в контейнерах.
- **Прод:** отдельный compose-оверрайд — фронт собран в статику (с service worker)
  и раздаётся nginx, реальные env, тома, TLS.

### 5.3. Порядок: от локалки к VPS
Сервер покупается в последнюю очередь. Вся инфраструктура обкатывается на своём
железе, упаковывается в Docker, на VPS разворачивается готовый механизм
(«склонировать репо → `up`»). Подробно — в разделе 8.

## 6. Безопасность

Разложено по типам угроз; для каждого — чем закрываемся. Для чат-платформы с медиа
это критично, поэтому вынесено в отдельный раздел.

### 6.1. Аутентификация и аккаунты
Угрозы: слабые пароли, перебор, кража токена. Защита: пароли только хешируем
(bcrypt/argon2, никогда plaintext); rate-limit на вход (Redis); короткий access +
отзываемый refresh в Redis; всё по HTTPS; аккуратное хранение токенов на фронте
(refresh — в httpOnly-cookie, access — в памяти приложения).

### 6.2. Авторизация — приоритет №1 для чата
Угроза: участник лезет туда, куда нельзя — читает чужой личный чат или дёргает
админскую функцию, подменив `room_id` в запросе (класс **IDOR / broken access
control**, самый частый класс веб-дыр). Защита: на **каждом** запросе сервер
проверяет, что юзер реально состоит в комнате / имеет нужную роль. Клиенту не
доверяем никогда. Любое чтение сообщений и любое действие в комнате проверяет
доступ серверно.

### 6.3. Инъекции и XSS
- **SQL-инъекции** закрывает SQLAlchemy (параметризованные запросы, не клеим SQL
  из строк).
- **XSS:** сообщение/материал с `<script>` не должен исполняться в чужих браузерах.
  React по умолчанию экранирует; markdown в сообщениях и базе знаний — обязательно
  **санитизировать** при рендеринге.

### 6.4. Загрузка файлов
Угрозы: вредоносный файл, подделка типа, гигантский файл (DoS), исполняемое под
видом картинки. Защита: бэкенд валидирует тип, реальное содержимое и размер **до**
выдачи presigned-PUT (лимиты в самой подписи); бакеты MinIO приватны, прямой доступ
только по presigned-ссылкам; имена объектов генерируем свои, не доверяя
пользовательским; заголовки отдачи запрещают исполнение.

### 6.5. WebSocket
Угрозы: неаутентифицированные подключения, спам, подписка на чужие комнаты. Защита:
проверять JWT при рукопожатии сокета; проверять членство перед подпиской на
комнату; rate-limit сообщений через Redis; только WSS (шифрованный).

### 6.6. Rate-limiting и DoS
Флуд сообщениями, долбёжка API, спам-регистрации. Защита: лимиты на юзера/IP
(Redis), лимиты на уровне nginx, при необходимости — капча на регистрацию.

### 6.7. Инфраструктура и секреты
Самая частая фатальная ошибка. Секреты (пароль БД, **секретный ключ JWT**, ключи
доступа MinIO) — только в переменных окружения, `.env` в `.gitignore`, **никогда в
git** (утёк ключ JWT = кто угодно подделает админский токен). Postgres, Redis и
MinIO **не публиковать наружу** — живут только в docker-сети; наружу торчит только
nginx. Регулярно обновлять ОС и зависимости.

### 6.8. Данные и приватность
Бэкапы Postgres (и безопасность, и доступность). Проект коммерческий и хранит
персональные данные (email, переписка): в РФ может быть применим **152-ФЗ** о
персональных данных — проверить на момент запуска (это не юридическая консультация,
а флажок).

## 7. Процесс разработки

### 7.1. Git-стратегия (Git Flow)
- `main` (`prod`) — всегда то, что в проде. Прямой пуш запрещён.
- `develop` — интеграционная ветка, «следующий релиз».
- `feature/*` — по ветке на фичу, от `develop`, вливается через PR.
- `hotfix/*` — срочные правки от `main`, вливаются в `main` и `develop`.
- Branch protection: обязательный PR + зелёный CI перед влитием в `main`/`develop`.

### 7.2. CI/CD (GitHub Actions)
- **CI** (линт, тесты, сборка на каждый PR) — в облаке GitHub, **сервер не нужен**.
- **CD** (доставка на хост по влитию в `main`) — единственная часть, требующая
  сервера. Поэтому VPS — финальные ~10% работы.

### 7.2a. Blue-Green деплой (zero-downtime)
Два одинаковых окружения приложения: **blue** (живой) и **green**. Новая версия
катится в green, проверяется, затем nginx переключает трафик blue → green. Blue
остаётся как мгновенный откат; следующий деплой роли меняются.

Три обязательных условия для нашего стека:
- **Stateful-сервисы не дублируются.** Удваиваются только stateless API и фронт.
  **Postgres, Redis и MinIO — в одном экземпляре**, общие для blue и green. Два
  Postgres не поднимаем никогда.
- **Миграции обязаны быть обратно-совместимыми (паттерн expand/contract).** База
  общая → в момент переключения и при откате старый и новый код работают с одной
  схемой. Нельзя в одной миграции переименовать/удалить колонку. См. 7.5.
- **WebSocket-дренаж.** При гашении blue его сокеты рвутся; клиент должен
  переподключаться к green (реконнект он умеет в любом случае). На нашем масштабе
  короткий блип переподключения приемлем; при желании — держать blue живым до
  миграции соединений.

> Трезвая оговорка: на 20–30 пользователях blue-green — лёгкое золочение; почти
> тот же эффект даёт rolling-restart. Решение принято осознанно; основная его цена
> — дисциплина обратно-совместимых миграций навсегда.

### 7.3. Параллельная разработка через Claude Code
- **git worktrees** — отдельная рабочая папка под каждую ветку. Связка:
  **worktree ↔ feature-ветка ↔ сессия Claude Code ↔ Pull Request**, каждый PR
  независимо гоняется через CI. Параллельные сессии не правят одни и те же файлы.
- При необходимости — субагенты и оркестрация (`/batch`) поверх worktree.

### 7.4. Память Claude между сессиями
- **`CLAUDE.md`** — автоматически в контексте каждой сессии: архитектура,
  соглашения, решения. Версионируется с кодом, общий для всех worktree.
- **Auto memory** — накапливающийся контекст между сессиями.
- Лог решений (`DECISIONS.md` / `docs/`).
> Это память агента-разработчика. История чатов пользователей — отдельно, в
> Postgres; её «выгрузка вовне» — это бэкапы базы.

### 7.5. Миграции (Alembic)
- Любое изменение структуры БД — файл-миграция с `upgrade`/`downgrade`. Цепочка
  миграций = история схемы (как git для таблиц).
- Миграции в git вместе с кодом; на проде применяются одной командой. Руками
  таблицы не меняются.
- **Только обратно-совместимые миграции (expand/contract)** — требование blue-green
  (см. 7.2a): сначала добавляем (expand) и катим код; удаляем старое (contract)
  отдельным релизом, когда старый код уже не используется. Переименование колонки =
  добавить новую + копировать + позже удалить старую, а не `RENAME` в один шаг.

## 8. Порядок работ

| Шаг | Что делаем | Нужен ли сервер |
|-----|------------|-----------------|
| 1 | Репозиторий + git-скелет (`main`, `develop`, branch protection). Монорепо. | Нет |
| 2 | `CLAUDE.md` + контекст проекта + лог решений. | Нет |
| 3 | Локальный compose: Postgres + Redis. | Нет |
| 4 | Модель данных + SQLAlchemy-модели + Alembic-миграции (`DATA_MODEL.md`). | Нет |
| 5 | Скелет бэкенда: auth (JWT), REST-эндпоинты, затем WebSocket-слой. API в compose. | Нет |
| 6 | CI на GitHub Actions (линт, тесты, сборка на PR). | Нет |
| 7 | Фронтенд (React/Vite) по дизайн-системе + сборка как PWA. Подключение к API. | Нет |
| 8 | Полная докеризация + прод-конфиг compose (nginx, env, тома, медиа). | Нет |
| 9 | **Покупка VPS.** Ubuntu + Docker, blue-green CD, домен, TLS. | Да |
| 10 | Эксплуатация: бэкапы Postgres в объектное хранилище, мониторинг. | Да |

К шагу 9 на руках полностью рабочая система, отлаженная локально и прогоняемая
через CI. Сервер — финальная коробка для готового механизма.

## 9. Открытые вопросы и статус решений

**Открытые вопросы (детали реализации):**
- Состав полей личного кабинета и `users.settings`.
- Точная схема хранения медиа и момент перехода на объектное хранилище.
- Кто и как добавляет стикерпаки, формат стикеров.
- Хранение токенов на фронте (httpOnly-cookie + защита от CSRF) — финал при реализации.

**Решено (зафиксировано):**
- Аутентификация — JWT (access + refresh), refresh-токены в Redis (отзыв/логаут).
- Уроки как сущности отсутствуют — вместо них база знаний под управлением админа.
- Треды одноуровневые (`thread_root_id`).
- Каналы — неявный доступ (вариант А).
- Мягкое удаление сообщений (`deleted_at`).
- Закрепление сообщений — отдельная таблица, несколько на комнату.
- «Печатает» и presence — эфемерно в Redis, не в БД.
- Прочтения — через `last_read_message_id`, без отдельной таблицы.
- Аватарки комнат — для групп и каналов.
- Привязка событий календаря к комнатам/каналам — да.
- Видео на старте раздаётся через nginx.
- Мобильный доступ — PWA (без нативных приложений и сторов).
- Redis подключается сразу; доставка через pub/sub с первого дня.
- Медиа — в **MinIO** (S3-совместимо), приватный доступ через presigned URL;
  загрузка/чтение напрямую клиент↔MinIO, минуя FastAPI.
- Деплой — **blue-green** (zero-downtime); stateful-сервисы общие, миграции
  обратно-совместимые (expand/contract).

**На вырост (вне MVP):**
- Пуш-уведомления (Web Push): хранение подписок, отправка, ограничения iOS.
- Категории базы знаний (`kb_categories`).
